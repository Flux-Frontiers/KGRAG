# Forest of Knowledge: Fractal Tree Visualizer — Design Plan

## Vision

Each registered knowledge graph is rendered as a fractal tree.  Its visual
properties — size, branching geometry, colour, species — are a direct encoding
of its graph structure.  Multiple trees form a forest.  Person corpora form
groves of closely spaced trees.  The aggregate of all registered KGs is the
**Tree of Knowledge(tm)**: the entire compiled corpus, made visible.

Running a federated query lights up branches across the forest in real time.
Replaying snapshot history animates the forest through time: branches grow,
colours deepen, new saplings appear as new KGs are registered.

---

## Data Model: Graph Properties → Tree Properties

Every tree is parameterised from `adapter._graph_stats()` plus `adapter.stats()`
and the `KGEntry` metadata.

| Source property | Tree parameter | Notes |
|---|---|---|
| `node_count` | Height, overall scale | Log-scaled so trees remain visible at all sizes |
| `edge_count / node_count` | Branching factor per level | Dense graphs → bushy canopy |
| `KGKind` | Species (trunk shape + angle distribution) | See species table below |
| `stats["status"]` | Opacity | Unavailable KGs render as ghost-pale outlines |
| Snapshot delta `Δnode_count` | New-growth colour at tips (fresh green) | Only when snapshot history is loaded |
| Coverage score (if available) | Colour saturation | Low coverage → desaturated / grey-green |
| `entry.created_at` | Bark texture weight | Older KGs appear more gnarled |
| `entry.tags` | Leaf shape | Tags map to a small set of leaf archetypes |

### Species by KGKind

| Kind | Species model | Visual character |
|---|---|---|
| `code` | Conifer / spruce | Dense, symmetrical, deep branches, dark green |
| `doc` | Broad-leaf deciduous | Wide canopy, irregular, light green |
| `diary` | Gnarled oak | Asymmetric, temporal, warm amber-brown |
| `meta` | Lattice / fern | Recursive, loop-tolerant, teal-blue |
| `verse` | Weeping willow | Long drooping branches, silver-grey |
| `memory` | Birch | Pale, sparse, vertical clusters |
| `disulfide` | Vine / tangle | Intertwined, short internode distance |
| `pdbfile` | Crystal / dendrite | Angular, symmetric, gold |
| `legal` | Columnar poplar | Tall, strict vertical hierarchy, deep burgundy |
| `person` | Bonsai (composite) | Compact, multi-stem, mixed foliage from sub-KGs |

## Fractal Tree Algorithm

Each tree is generated by a **stochastic L-system** with parameters drawn from
the KG's graph properties.  The algorithm is applied recursively:

```
draw_branch(pos, angle, length, depth, branching_factor, spread_angle, seed):
    if depth == 0: draw_leaf(); return
    draw_line(pos, angle, length)
    new_pos = pos + (length * cos(angle), length * sin(angle))
    n_children = sample_poisson(branching_factor)
    for i in range(n_children):
        child_angle = angle + spread_angle * (i - n_children/2) + jitter(seed)
        child_length = length * length_ratio + jitter(seed)
        draw_branch(new_pos, child_angle, child_length, depth-1, ...)
```

Parameters:
- `depth`: `ceil(log2(node_count))` — deeper graphs produce taller trees
- `branching_factor`: `edge_count / node_count` clamped to `[1.2, 4.0]`
- `spread_angle`: species-specific base angle (e.g. 30° for conifers, 60° for oaks)
- `length_ratio`: 0.65–0.75, species-specific
- `seed`: derived from `KGEntry.id` for reproducible rendering

## Forest Layout

### Single-KG forest

Each KG occupies a position in a 2D plane computed by a force-directed layout
over the KG graph — KGs that share tags or corpus membership are placed closer
together.  Minimum spacing prevents canopy overlap.

### Person corpus grove

All KGs belonging to a `PersonCorpusEntry` are placed within a bounded ellipse,
their trunks radiating outward from a shared centroid.  The diary KG is always
the largest and most central.  A soft glow effect marks the grove boundary.

### Corpus groups

Named `CorpusEntry` groups render as a loose ring of trees with a subtle ground
fog at their bases, visually distinguishing them from ungrouped singletons.

### Tree of Knowledge(tm)

All registered KGs rendered simultaneously at a reduced zoom level.  Person
groves appear as dense clusters; ungrouped KGs stand in clearings between them.
A global illumination pass adjusts ambient brightness by corpus activity (more
recent `updated_at` → brighter ambient).

## Query Highlighting

When a federated query is executed, the visualizer receives the `CrossQueryResult`
and maps each `CrossHit` to its source KG tree:

1. The tree(s) that returned results **pulse** — a ripple of light from the
   trunk outward through the branches.
2. Individual branches corresponding to the hit's node type are **illuminated**
   in query-gold for the duration of the result display.
3. The brightness of illumination scales with the hit's relevance score.
4. KGs that returned no results dim slightly to guide attention.

This creates a live "which trees know about this?" readout that makes
cross-domain connections immediately visible.

## Temporal Rendering via Snapshots

When snapshot history is available for a KG, the visualizer can replay its
growth:

```
timeline_play(kg_name, speed=1.0):
    for snapshot in snapshot_list(kg_name):          # chronological order
        target_params = params_from_snapshot(snapshot)
        tween(current_tree, target_params, duration=1.0/speed)
        current_tree = target_params
        sleep(frame_duration / speed)
```

Each snapshot transition:
- Adjusts tree height and branching density smoothly
- Sprouts new-growth tips (bright green) for added nodes
- Fades branches (amber decay) for removed nodes
- Updates the colour saturation to reflect coverage change

For the full forest, all KGs can animate together, showing the global
knowledge base growing and changing over time.

## Implementation Plan

### Phase 1 — Static single-tree renderer (MVP)

**Goal:** Given a `KGEntry` + `_graph_stats()`, render one tree to a PNG/SVG.

- [ ] `TreeParams` dataclass: encode all visual parameters
- [ ] `kg_to_tree_params(entry, stats) -> TreeParams`: mapping function
- [ ] `FractalTreeRenderer.draw(params) -> PIL.Image`: recursive L-system renderer
- [ ] CLI: `kgrag viz-forest <kg-name> --out tree.png`
- [ ] Unit tests for param derivation (known node/edge counts → expected height)

**Technology:** Python, `Pillow` for raster output, `svgwrite` for SVG.
No runtime dependencies beyond what KGRAG already uses.

**Deliverable:** `src/kg_rag/visualizer/tree.py` + `src/kg_rag/visualizer/renderer.py`

---

### Phase 2 — Multi-tree forest layout

**Goal:** Render the full registered forest to a single image.

- [ ] `ForestLayout.place(entries) -> dict[str, (x, y)]`: force-directed placement
- [ ] Grove detection from `PersonCorpusRegistry` and `CorpusRegistry`
- [ ] Composite canvas: place each tree at its assigned position
- [ ] CLI: `kgrag viz-forest --out forest.png`
- [ ] Legend overlay: KG name, kind, node count

**Deliverable:** `src/kg_rag/visualizer/forest.py`

---

### Phase 3 — Interactive real-time renderer

**Goal:** Live window with query highlighting and pan/zoom.

- [ ] Port renderer to `pyglet` or `pygame` for GPU-accelerated 2D
- [ ] Glow shader for illumination effects (GLSL fragment shader via pyglet)
- [ ] Query callback: accepts `CrossQueryResult`, triggers branch illumination
- [ ] Pan/zoom controls (mouse drag, scroll wheel)
- [ ] Click on tree → show KG name, stats, snapshot delta in sidebar
- [ ] CLI: `kgrag viz-forest --interactive`

**Deliverable:** `src/kg_rag/visualizer/interactive.py`

---

### Phase 4 — Temporal animation

**Goal:** Replay snapshot history as animated forest growth.

- [ ] `SnapshotPlayer`: loads snapshot list, interpolates tree params
- [ ] Tween engine: smooth parameter transitions between snapshots
- [ ] New-growth particle effect on added nodes
- [ ] Timeline scrubber: click to jump to any snapshot
- [ ] Export to MP4/GIF: `kgrag viz-forest --animate --out growth.mp4`

**Deliverable:** `src/kg_rag/visualizer/animation.py`

---

### Phase 5 — Web / 3D rendering (future)

**Goal:** Browser-based 3D forest with full interactivity.

- [ ] Export tree params to JSON for JavaScript consumption
- [ ] Three.js 3D fractal tree renderer
- [ ] WebSocket bridge from KGRAG MCP server to browser
- [ ] VR-ready scene graph (A-Frame or Babylon.js)

## Open Questions

1. **Raster vs vector for Phase 1?** SVG scales better for print/zoom; PNG
   simpler for MVP. Recommend SVG via `svgwrite`, with PNG export option.
2. **Determinism?** Tree shape should be reproducible from `KGEntry.id` as seed.
   Confirmed via `params.seed = hash(entry.id) % (2**32)`.
3. **Maximum tree count?** Force layout degrades beyond ~200 nodes.  For very
   large registries, cluster KGs before layout and show cluster representatives.
4. **3D vs 2D for Phase 3?** 2D with glow effects achieves the visual goal with
   simpler implementation.  3D adds depth cues but requires more infrastructure.
   Start 2D, promote to 3D in Phase 5.
